== Purpose

This document describes the detailed design of the AS Audio Service configuration module (`AS_AudioService`) implemented under `hlos_qnx_8255/apps/qnx_ap/AMSS/multimedia/audio/audio_ar/stla_audio/AS_AUDIO/AS_AudioService`.

The module provides Audio Service runtime behavior for:

* QNX IPC server-side request handling (`name_attach`, `MsgReceive`, `MsgReply`)
* Audio playback control (create/start/stop/get status)
* Volume and speaker-channel mute control through CSD2 post-processing
* Audio recording start/stop handling (when `AUDSER_SOUND_RECORD_ENABLE` is enabled)
* Event IPC initialization and status-notification response path for clients

This document aligns with:

* Current source implementation in `AS_AudioService` and associated `IAudSerIpc`
* Build/runtime integration (`common.mk`, CSD2 client, IPC library)
* Reference document `docs/Audio+Service+Implementation+details.pdf`

== General

=== Summary

Main goal of AS_AudioService_config is to provide deterministic, state-driven audio playback/capture control and runtime reconfiguration over IPC and CSD2 for internal amplifier use-cases.

The module abstracts app-type-dependent request handling, CSD2 transport/setup, and playback state transitions (Idle, Start, Starting, Failed). It aligns with platform audio architecture and supports dynamic playback control, volume operations, capture control, and event publishing.

Major responsibilities:

* Service lifecycle init/destroy
* Audio player instance management for supported app types
* Playback pipeline setup and data streaming
* Chime volume/angle and speaker mute management through CSD2 PP control
* Optional audio capture workflow
* IPC event channel setup and status event forwarding

=== Limitations on Usage

* Target OS/platform: QNX Neutrino on SA8255.
* Service endpoint name is fixed to `IAudser_SndIpc`.
* App-type capacity is compile-time constrained:
  - Internal amp mode (`AUDSER_INTERNAL_AMP_TYPE`): 2 app types (`enCcsChimeType`, `enIviChimeType`).
  - External amp mode: 1 app type.
* Playback channel mapping supports predefined channel counts (1/2/3/4/6/8/16/32).
* Audio format path is PCM-focused (`audSer_enAudioFormat::enAudioFormatPCM`).
* CSD2 and dependent libraries must be available (`csd2IpcClient`, `IAudSerIpc`, `slog2`, `bmetrics`).
* No direct GPIO device access/configuration exists in this module; hardware pin configuration is handled by lower-level audio driver/platform layers.

== SW Detailed Design

=== Static View

==== Interfaces of SW Units

The module is split into two major layers:

* Adapt Layer
  - `main.cpp`: process entry and signal mask setup
  - `AudioService_service1.*`: QNX IPC server loop and method dispatch
* Core Layer
  - `AudioService_AudSerInterface.*`: functional orchestration API
  - `AudioService_AudioPlayer.*`: app-type player control and thread launch
  - `AudioService_AudioPlayback.*`: CSD2 stream configuration and write loop
  - `AudioService_VolManagement.*`: volume and angle control through PP handle
  - `AudioService_SpkConfig.*`: speaker mute/unmute channel mask control
  - `AudioService_SoundRecord.*`: optional capture workflow
  - `AudioService_Csd2Interface.*`: CSD2 wrapper/helper functions
  - `AudioService_WavExtractor.*`: WAV header parse/write helper

==== GPIO PinMux Configuration

Implementation evidence shows that `AS_AudioService` performs no direct GPIO operations:

* No `/dev/gpio*` access in service/core source.
* No GPIO register/pinmux APIs called from `AS_AudioService` modules.
* Hardware routing and pin-level behavior are delegated to audio driver/board configuration layers outside this module.

Therefore, in this module context, GPIO configuration is an external dependency and not an internal design responsibility.

=== Dynamic View

==== Initialization Sequence

1. `main()` invokes `InitializeSignalHandling()`.
2. `AudSer_Service` object construction calls `audioService_vAudServiceInit()`.
3. `audioservice_run()` calls `name_attach(NULL, "IAudser_SndIpc", 0)`.
4. Service enters `while (bexit)` receive loop (`MsgReceive`).

==== Request Handling Sequence

1. Client sends method payload (`audSer_strMetaData_type`) through IPC.
2. Service receives rcvid and request union.
3. Service switches on method enum and calls corresponding `AudSer_AudSerInterface` API.
4. Service returns response via `MsgReply`.

==== State Machine

The runtime player state follows `audSer_enAudioPlayerStatus` transitions:

* `enAudioIdle` -> `enAudioStart` when playback start is accepted.
* `enAudioStart`/`enAudioStarting` -> `enAudioIdle` when playback completes successfully.
* `enAudioStart`/`enAudioStarting` -> `enAudioFailed` on playback/CSD2 error.
* Any connected client disconnect pulse forces stop handling and cleanup for mapped app type.

==== Playback Runtime Sequence

1. `AUDIOSTARTPLAYREQ` arrives with file path, app type, volume, chime angle, iteration count.
2. Interface validates player state.
3. Audio player updates payload to `enAudioStart` and launches thread.
4. Playback thread calls `audioService_u32StartAudioPlay(...)`.
5. Playback configures CSD2 session and stream format.
6. WAV data is written in chunks through CSD2 write loop.
7. On completion/failure, status is updated and PP handle is closed.

==== Disconnect and Cleanup Behavior

When `_PULSE_CODE_DISCONNECT` is received:

1. Service finds mapped app type for disconnected client.
2. Service triggers regular stop mode.
3. Event IPC channel for app is deinitialized.
4. Service clears mapping and detaches connection (`ConnectDetach`).

==== Event Notification Path

1. Client requests `AUDIOEVENTIPCREQ`.
2. Service initializes event IPC via `audioService_s32InitEventIpc(...)`.
3. Client side `IAudSerIpc` receives event payloads.
4. `enAudioOnStatusEvent` response content is forwarded to client callback/reply path.

=== Detailed Level Design

==== Interoperability

* Service loop is single dispatch loop; playback work is offloaded to detached threads per app type.
* Mutexes are used in player and volume/speaker modules for state/PP handle protection.
* Stop mode (`quick`/`regular`) is tracked in player state.
* App occupancy and iteration counts are maintained in player payload.

===== App Type Enumeration (Internal Amplifier Scope)

Reference enum and current design intent are aligned:

* `INVALID_CHIME_TYPE = 0`
* `CCS_CHIME_TYPE`
* `IVI_CHIME_TYPE`
* `HIGH_CHIME_TYPE`

In current implementation, internal amplifier mode is considered and active app types are mapped to service instances/graph IDs.

===== Initialization and Lifetime Behavior

* `onInitialize()`/`onDestroy()` are expected to be called once per server lifecycle (not per client application).
* Initialization flow includes create-instance preparation from service startup path.
* In internal-amp mode, two logical instances are supported:
  - Instance-1: Graph ID 26
  - Instance-2: Graph ID 50
* Allocation behavior is effectively first-available/first-served from service perspective.

===== Request/Response Message Pattern

All major playback control APIs use a common response payload concept (`WavPlayResponseMsg` in reference notes), represented in current module by `audioService_strAudPlaybackRes_type`.

Response fields conceptually include:

* app type
* request identifier
* response status
* repetition/iteration count
* playback status
* volume level

===== API-specific Behavior Notes

* Create instance must be completed before start playback for a given app type.
* During ongoing playback, client is expected to be able to call:
  - Set volume
  - Get volume
  - Stop playback
  - Get status
* Start capture is non-blocking and stop capture can be called at any time.
* Capture start validates input parameters, updates graph/device descriptors, and sets capture event registration path.

===== Event Publishing Semantics

Reference event outputs are aligned to response categories:

* Create response
* Start playback response
* Stop playback response
* Get status response
* Set volume response
* Get volume response
* Status event notification

Current design uses event IPC registration (`AUDIOEVENTIPCREQ`) and status event forwarding (`enAudioOnStatusEvent`) for this purpose.

===== Scope Notes from Implementation Details

* Internal amplifier use-case is the considered scope.
* InterVM behavior is not considered.
* Public event transport details from server to client remain a design decision item.

==== State Machine Design

For internal amp mode, graph mapping is explicitly assigned in player preparation:

* App type 1 (`enCcsChimeType`) -> Graph ID 26
* App type 2 (`enIviChimeType`) -> Graph ID 50

This mapping is used by playback and PP control data paths.

==== Volume, Angle, and Mute Handling

* Volume API tracks logical levels (Low/Mid/High).
* AWE path supports combined update: volume and chime angle.
* Speaker channel mute/unmute is encoded into bitmasks and sent via PP control.
* PP handle is closed on both success and failure paths to avoid stale handle leakage.

==== Recording Path

When recording is enabled by compile macro:

* Start/stop capture methods are accepted over IPC.
* Capture graph and callback path are configured in `AudioService_SoundRecord`.
* WAV header and file handling use `AudSer_WavExtractor` helper.

==== Error Handling

* QNX IPC failures (`MsgReceive`, `MsgReply`, `ConnectDetach`) are logged.
* CSD2 operation return values are checked and propagated.
* Null checks and allocation checks are present before processing.
* Build includes `slog2` and `bmetrics`; source includes detailed runtime `cout` diagnostics.

==== Criticality Analysis

* Service availability is critical for chime playback and event reporting.
* IPC loop robustness and deterministic cleanup on client disconnect are key runtime priorities.
* Failure impact is limited to audio service behavior and does not include direct GPIO or low-level pin control.

==== Complexity Analysis

* Moderate control-flow complexity due to IPC dispatch, multi-request handling, and threaded playback path.
* Data complexity is moderate and largely bounded by fixed payload types and enums.
* Integration complexity is medium-to-high because behavior depends on CSD2 driver stack and client IPC contract.

==== Testability

* Unit-level testability exists for enum/state handling and payload preparation.
* Integration testing is required for IPC request/response and CSD2 stream operations.
* Fault-injection test scenarios should include CSD2 failures, invalid app type, and disconnect pulses.

==== Risk Assessment

* Risk: malformed payload combinations from client.
  - Mitigation: enum/status validation and fail responses.
* Risk: stale PP/session handles across abnormal stop paths.
  - Mitigation: explicit close calls on success and failure paths.
* Risk: dependency failure in CSD2 or IPC attachment.
  - Mitigation: return-code checks, logging, and controlled service exit.

=== Restrictions and Constraints

* Service name and IPC contract are fixed by `IAudSerIpc` method definitions.
* Supported app types and graph mappings are compile-time constrained by amp mode.
* PCM playback path assumptions apply to current implementation.
* This module does not configure GPIO and relies on lower-layer platform/audio-driver setup.

=== Diagnosis and Logging

* Runtime logs are emitted on IPC receive/reply failures and disconnect handling.
* CSD2 return values are logged on configuration and stream operations.
* Build links `slog2` and `bmetrics` to support diagnostics/performance instrumentation.

=== Safety Assessment

* Service behavior is bounded to fixed IPC methods and typed payload unions.
* No direct privileged GPIO/pinmux writes are performed in this module.
* Residual safety risk remains around invalid client parameters and dependency faults.

=== Security Assessment

* Service is process-isolated and does not execute arbitrary code paths from IPC payloads beyond fixed method switch.
* Signal mask setup blocks most signals to reduce asynchronous interruptions.
* Interface uses fixed-size payload structures; callers must supply valid enumerations/parameters.
* No direct GPIO or privileged register writes are performed by this module.
* Residual risk: malformed client payload values can still request invalid combinations; robustness relies on downstream validation and response status handling.
